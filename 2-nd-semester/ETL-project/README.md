## 1. Что такое ETL-конвейер и зачем он нужен

ETL-конвейер — это автоматизированный процесс обработки данных, который включает три этапа:

1. **Extract (Извлечение)** — сбор данных из различных источников
2. **Transform (Трансформация)** — очистка, преобразование и обогащение данных
3. **Load (Загрузка)** — сохранение обработанных данных в хранилище для анализа

В рамках данной работы ETL-конвейер решает задачу автоматического сбора и анализа данных о дефектах оборудования ТЭЦ. Раньше такие данные обрабатывались вручную в Excel, что занимало много времени и приводило к ошибкам. Автоматизация позволяет:
- Получать актуальную статистику ежедневно
- Исключить человеческий фактор
- Быстро выявлять проблемные зоны
- Принимать обоснованные управленческие решения

## 2. Общая архитектура решения

Система построена по микросервисной архитектуре — каждый компонент выполняет свою функцию и работает независимо. Все компоненты запускаются в Docker-контейнерах, что обеспечивает:
- Изолированность сервисов
- Простоту развертывания
- Воспроизводимость среды
- Возможность масштабирования

### 2.1. Компоненты системы

| Компонент | Функция |
|-----------|---------|
| **PostgreSQL** | Хранит метаданные Airflow: историю запусков, статусы задач, настройки соединений |
| **ClickHouse** | Аналитическое хранилище, куда загружаются обработанные данные о дефектах |
| **Apache Airflow** | Оркестратор — планирует и выполняет ETL-процессы по расписанию |
| **Apache Superset** | Инструмент визуализации — создает дашборды и графики на основе данных |

### 2.2. Поток данных

1. Исходный Excel-файл с данными о дефектах помещается в папку `data`
2. Airflow по расписанию (ежедневно в 8:00) запускает ETL-процесс
3. Данные извлекаются из Excel, очищаются и преобразуются
4. Обработанные данные загружаются в ClickHouse
5. Superset подключается к ClickHouse и отображает данные на дашбордах

## 3. Этап 1: Извлечение данных (Extract)

На этом этапе данные загружаются из источника. Источником служит Excel-файл, который содержит информацию о дефектах оборудования ТЭЦ.

### 3.1. Исходные данные

Excel-файл содержит следующие основные поля:

| Поле | Описание |
|------|----------|
| unique_number | Уникальный номер дефекта |
| filial | Филиал организации |
| station | Конкретная станция/ТЭЦ |
| priority | Приоритет дефекта (1 — самый критичный, 4 — наименее критичный) |
| alias | Статус дефекта (ELIMINATED — устранен, REJECTED — отклонен и др.) |
| created_date | Дата создания заявки |
| Плановый_срок_устранения | Плановая дата устранения |
| Фактический_срок_устранения | Фактическая дата устранения |

### 3.2. Особенности извлечения

- Данные считываются построчно из Excel
- Сохраняется вся информация без изменений
- Формат данных (строки, даты, числа) пока не приводится к единому стандарту
- На этом этапе не выполняется никаких преобразований

---

## 4. Этап 2: Трансформация данных (Transform)

Это самый важный и сложный этап. Данные приводятся к единому формату, очищаются от ошибок и обогащаются новыми расчетными показателями.

### 4.1. Переименование и стандартизация полей

Все колонки переименовываются в единый стандарт (англоязычные названия без пробелов), чтобы упростить дальнейшую работу с данными. Например:
- `filial` → `branch`
- `unique_number` → `defect_id`
- `alias` → `status`

### 4.2. Преобразование дат

Даты приводятся к единому формату. Это важно, потому что в исходном Excel даты могут быть записаны в разных форматах (день/месяц/год или месяц/день/год). В процессе преобразования:
- Все даты становятся объектами типа `Date`
- Некорректные даты помечаются как пустые значения (`NULL`)
- Ошибки преобразования не останавливают весь процесс

### 4.3. Преобразование числовых значений

Приоритет дефекта преобразуется из текстового формата в целое число. Если значение не удается преобразовать, оно заменяется на ноль.

### 4.4. Расчет новых метрик

На основе исходных данных рассчитываются новые показатели:

| Метрика | Формула | Бизнес-смысл |
|---------|---------|--------------|
| fix_time_days | фактическая дата устранения − дата создания | Время устранения дефекта в днях |
| overdue_days | фактическая дата устранения − плановая дата | Просрочка устранения в днях |
| is_overdue | 1 если overdue_days > 0, иначе 0 | Был ли дефект устранен с просрочкой |
| is_plan_met | 1 если устранено до плановой даты, иначе 0 | Соблюдены ли плановые сроки |

### 4.5. Категоризация дефектов

По текстовому описанию дефекта определяется его категория. Это позволяет группировать дефекты по типам для дальнейшего анализа:

| Категория | Ключевые слова в описании |
|-----------|---------------------------|
| leak (течь) | "течь", "свищ", "парение" |
| seal (сальник) | "сальник" |
| valve (арматура) | "задвижка", "вентиль", "арматура" |
| electrical (электрика) | "электрический", "кабель", "освещение" |
| other (прочее) | Все остальное |

### 4.6. Очистка данных (Data Cleaning)

Выполняется фильтрация и очистка данных:
- Удаляются записи с некорректной датой создания
- Оставляются только устраненные дефекты (статус ELIMINATED)
- Удаляются выбросы:
  - Время устранения не может быть отрицательным
  - Время устранения не может превышать 1000 дней
  - Просрочка ограничивается значениями от -1000 до 1000 дней
- Пустые значения заменяются на осмысленные (ноль для чисел, пустая строка для текста)

## 5. Этап 3: Загрузка данных (Load)

Обработанные данные загружаются в ClickHouse — колоночную аналитическую СУБД, оптимизированную для быстрых агрегатных запросов.

### 5.1. Почему ClickHouse?

ClickHouse выбран потому что:
1. **Высокая скорость агрегаций** — запросы на миллионах строк выполняются за миллисекунды
2. **Эффективное сжатие** — данные сжимаются в 5-10 раз, экономя дисковое пространство
3. **Поддержка SQL** — стандартный язык запросов, легкость освоения
4. **Интеграция с Superset** — нативная поддержка через драйвер

### 5.2. Структура таблицы

В ClickHouse создается таблица `defect_analysis` со следующей структурой:
- Идентификаторы: defect_id, branch, station, department
- Текстовые поля: equipment_name, defect_description, status, responsible_person
- Числовые поля: priority, fix_time_days, overdue_days, is_overdue, is_plan_met
- Даты: created_date, planned_deadline, actual_fix_date
- Категория: defect_category

Таблица использует движок `MergeTree`, который сортирует данные по полям `branch` и `created_date`. Это ускоряет запросы по этим ключевым полям.

### 5.3. Процесс загрузки

Данные загружаются пакетами по 500 записей для оптимизации производительности:
1. Устанавливается соединение с ClickHouse
2. Проверяется существование таблицы (если нет — создается)
3. Данные преобразуются в формат, понятный ClickHouse
4. Выполняется пакетная вставка
5. В случае ошибки в пакете — записи вставляются по одной, проблемные пропускаются

## 6. Оркестрация с Apache Airflow

### 6.1. Что такое DAG

DAG (Directed Acyclic Graph) — это ориентированный ациклический граф, который описывает последовательность задач, их зависимости и расписание выполнения.

В рамках проекта создан один DAG с именем `tec_breakdown_etl`, который:
- Выполняется ежедневно в 8:00 утра
- Состоит из трех последовательных задач:
  1. Извлечение данных из Excel
  2. Трансформация данных
  3. Загрузка данных в ClickHouse

### 6.2. Зависимости между задачами

Задачи выполняются строго последовательно:
- Трансформация не может начаться, пока не завершено извлечение
- Загрузка не может начаться, пока не завершена трансформация

Если какая-то задача завершается с ошибкой, последующие не запускаются.

### 6.3. Мониторинг

Airflow предоставляет веб-интерфейс, в котором можно:
- Отслеживать статус выполнения каждого DAG
- Просматривать логи выполнения каждой задачи
- Вручную запускать DAG при необходимости
- Настраивать оповещения об ошибках

## 7. Визуализация в Apache Superset

### 7.1. Подключение к ClickHouse

Superset подключается к ClickHouse как к источнику данных. Настройка включает:
- Указание хоста (clickhouse) и порта (8123)
- Имя базы данных (defects_db)
- Проверку соединения

### 7.2. Созданные визуализации

| Визуализация | Тип | Что показывает |
|--------------|-----|----------------|
| Дефекты по филиалам | Столбчатая диаграмма | В каком филиале больше всего дефектов |
| Динамика по месяцам | Линейная диаграмма | Изменение количества дефектов во времени |
| Дефекты по приоритетам | Столбчатая диаграмма | Распределение по степени критичности |
| Дефекты по станциям | Горизонтальная столбчатая | Какая ТЭЦ требует особого внимания |
| Категории дефектов | Круговая диаграмма | Структура дефектов по типам |
| Статусы дефектов | Круговая диаграмма | Доля устраненных/неустраненных |

### 7.3. Дашборд

Все визуализации собраны в единый дашборд "Анализ дефектов ТЭЦ", который позволяет:
- Видеть общую картину по всем показателям
- Фильтровать данные по филиалам и станциям
- Анализировать динамику изменений
- Выявлять проблемные зоны

## 8. Контейнеризация с Docker

### 8.1. Зачем нужен Docker

Docker позволяет упаковать каждый компонент системы вместе со всеми его зависимостями в изолированный контейнер. Это дает:
- **Воспроизводимость** — система работает одинаково на любом сервере
- **Изоляцию** — конфликты между компонентами исключены
- **Переносимость** — можно запустить на любом сервере с Docker
- **Упрощение развертывания** — одна команда запускает все сервисы

### 8.2. Компоненты в контейнерах

Каждый сервис запускается в своем контейнере:
- PostgreSQL — официальный образ PostgreSQL 13
- ClickHouse — официальный образ ClickHouse 23.8
- Airflow — кастомный образ на основе официального Airflow 2.8.2
- Superset — кастомный образ на основе официального Superset 3.1.3

### 8.3. Docker Compose

Docker Compose — это инструмент для запуска многоконтейнерных приложений. Один файл `docker-compose.yml` описывает:
- Какие контейнеры нужно запустить
- Какие порты открыть
- Какие тома использовать для хранения данных
- Как контейнеры взаимодействуют друг с другом
- В каком порядке запускать контейнеры

### 8.4. Тома (Volumes)

Тома обеспечивают сохранность данных между перезапусками контейнеров:
- postgres-db-volume — данные PostgreSQL
- clickhouse-data — данные ClickHouse
- superset-data — данные Superset

## 9. Развертывание и запуск

### 9.1. Требования к системе

Для запуска проекта необходимо:
- Docker версии 24.0 или выше
- Docker Compose версии 2.20 или выше
- 4 ГБ оперативной памяти (рекомендуется 8 ГБ)
- 5 ГБ свободного места на диске

### 9.2. Процесс развертывания

1. Клонирование репозитория с проектом
2. Запуск команды `docker-compose up -d`
3. Ожидание запуска всех контейнеров (около 1-2 минут)
4. Проверка статуса через `docker-compose ps`

### 9.3. Доступ к сервисам

После развертывания доступны следующие сервисы:

| Сервис | URL | Логин | Пароль |
|--------|-----|-------|--------|
| Airflow | http://localhost:8080 | admin | admin |
| Superset | http://localhost:8088 | admin | admin |
| ClickHouse | http://localhost:8123 | default | - |
